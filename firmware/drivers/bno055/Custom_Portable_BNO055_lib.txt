## BNO055 Driver

A portable and hardware-agnostic driver for the BNO055 IMU, built using a layered architecture that cleanly separates sensor logic from platform-specific code.

No dependency on HAL, RTOS, or any specific framework — just provide I2C and delay functions.

## Design Philosophy

- No hardware dependency (fully portable)
- Clear separation of responsibilities
- Simple and predictable API

## Responsibilities

### Driver Responsibilities

- Sensor register communication (read/write)
- Data acquisition and parsing
- Mode control and configuration
- Platform-independent API

---

- Provide I2C read/write
- Provide delay function
- Handle system logic
- Initialize hardware

## Step 2: Initialize Driver

BNO055 supports two I2C addresses based on the state of the address pin:


0x28 → default (ADR pin LOW)

0x29 → ADR pin HIGH



```c
bno055_t imu;

bno055_io_t io = {
    .ctx = YOUR_I2C_CONTEXT,
    .i2c_read = i2c_read,
    .i2c_write = i2c_write,
    .delay_ms = delay_ms
};

bno_init(&imu, &io, 0x28);
```

**Important:**

### Recommended Initialization Checks

**What this section does:**  
Provides basic checks to ensure the sensor is correctly connected and configured before regular usage.



### Check Chip ID

Verify that the connected device is a valid BNO055.


```c
if (bno_check_chip_id(&imu) == BNO_OK)
{
    // Sensor detected correctly
}
else
{
    // Error: wrong device or communication failure
}
```


### Data Scaling

Sensor values are returned as raw integers and must be scaled:

- **Euler angles (yaw, roll, pitch):** divide by 16 → degrees  
- **Gyroscope (gx, gy, gz):** divide by 16 → °/s  
- **Accelerometer (ax, ay, az):** divide by 100 → m/s²  

---

## Step 4: Read Sensor Data
### Using the Sensor Data

The data is stored in the `bno_9val_t` structure:

```c
typedef struct
{
    int16_t yaw, roll, pitch;
    int16_t gx, gy, gz;
    int16_t ax, ay, az;
    int16_t qw, qx, qy, qz;
    int16_t gravx, gravy, gravz;
} bno_9val_t;
```

```c
bno_9val_t data;

if (bno_read_9val(&imu, &data) == BNO_OK)
{
    // Orientation (Euler angles)
    int16_t yaw   = data.yaw;
    int16_t roll  = data.roll;
    int16_t pitch = data.pitch;

    // Gyroscope (angular velocity)
    int16_t gx = data.gx;
    int16_t gy = data.gy;
    int16_t gz = data.gz;

    // Accelerometer (linear acceleration)
    int16_t ax = data.ax;
    int16_t ay = data.ay;
    int16_t az = data.az;
}
```

## Step 2: Implement IO Functions (HAL)

BNO055 supports two I2C addresses based on the state of the address pin:

0x28 → default (ADR pin LOW)
0x29 → ADR pin HIGH


```c
int stm32_i2c_read(void *ctx,
                   uint8_t addr,
                   uint8_t reg,
                   uint8_t *data,
                   uint16_t len)
{
    HAL_I2C_Mem_Read((I2C_HandleTypeDef*)ctx,
                     addr << 1,
                     reg,
                     I2C_MEMADD_SIZE_8BIT,
                     data,
                     len,
                     100);
    return 0;
}

int stm32_i2c_write(void *ctx,
                    uint8_t addr,
                    uint8_t reg,
                    uint8_t *data,
                    uint16_t len)
{
    HAL_I2C_Mem_Write((I2C_HandleTypeDef*)ctx,
                      addr << 1,
                      reg,
                      I2C_MEMADD_SIZE_8BIT,
                      data,
                      len,
                      100);
    return 0;
}

void stm32_delay(uint32_t ms)
{
    HAL_Delay(ms);
}
```

## Limitations

- Uses blocking I2C communication (no non-blocking or DMA support)  
- No interrupt-driven interface  
- No built-in calibration save/restore support  
- No burst-read optimization (multiple I2C transactions are used)  
- No SPI support (I2C only)  
- Uses user-provided IO functions for I2C and delay, enabling portability across different platforms
